# Gemini-generated synthetic label test set

**Status:** Plan (not built).
**Owner:** TTB-000 (this story packet).
**Trigger to build:** after the live COLA Cloud loader ships and we've used it enough to confirm we want seeded eval data too.

## Why

The `cola-cloud-all` corpus (28 real TTB-approved labels + declared
fields) is the gold set for accuracy evals. But it has two gaps:

1. **It's all real `approve` cases.** The pipeline's reject paths —
   ABV mismatch, missing warning, wrong-country import — never get
   exercised from live fixtures unless we hand-edit the CSV. That
   means we can't measure the pipeline's _true negative_ rate
   (correctly flagging bad labels) without manufactured cases.
2. **It's small and static.** 28 labels won't stress the batch
   pipeline, warning-vote edge cases, or rare beverage types. We want
   to scale eval coverage without scraping more COLA Cloud entries
   (which takes days at the free-tier rate limit).

Gemini's multimodal image generation can produce **photorealistic
label images** conditioned on a prompt. If we also record the
**intended defect** and the **expected verdict**, every synthetic
label becomes a ground-truth eval case.

## What gets built

### Server side

- New endpoint: `POST /api/eval/synthetic/generate`
  - Request: `{ beverageType: 'wine' | 'malt-beverage' | 'distilled-spirits', seed?: number, defect?: DefectKind }`
  - Pulls the Imagen model via the existing `@google/genai` SDK
    (same key as `GEMINI_API_KEY`).
  - Builds an instruction with the template prompt + realistic brand
    attributes (synthesized from a name generator) + the chosen defect
    class injected as a visual/textual instruction to the model.
  - Returns `{ image, declaredFields, expectedVerdict, defectKind }`.
  - The `expectedVerdict` becomes the golden label for any downstream
    eval.

- New catalogue file: `evals/synthetic/generator-manifest.json`
  - Tracks the last 100 generated labels so the frontend can replay a
    specific seed + see its diff against the pipeline's verdict.

### Defect kinds

Defects map 1:1 to the field-judge rules the pipeline enforces. Each
defect is a pair: (image-side mutation, declared-field mutation, expected verdict).

| DefectKind | Image says | Declared says | Expected verdict |
|---|---|---|---|
| `none` | matches declared | matches image | `approve` |
| `brand-typo` | "Stone's Throw" | "STONES THROW" | `approve` (fuzzy-close) |
| `brand-totally-different` | "Mill River Cider" | "Stone's Throw" | `review` |
| `abv-small-drift` | "40% Alc./Vol." | "40.3% Alc./Vol." | `approve` (rounding) |
| `abv-wine-tax-cross` | "14.5% Alc./Vol." | "13.5% Alc./Vol." | `reject` (crosses 14% boundary) |
| `abv-way-off` | "40%" | "38%" | `reject` |
| `net-fl-oz-rounding` | "25.4 fl oz" | "750 mL" | `approve` (within 5mL tolerance) |
| `net-mismatch` | "700 mL" | "750 mL" | `reject` |
| `warning-missing` | no warning text | standard | `reject` |
| `warning-key-phrase-missing` | warning without "during pregnancy" clause | standard | `reject` |
| `warning-noisy` | warning with OCR-style noise (15 char drift) | standard | `approve` (fuzzy ≥ 0.9) |
| `country-equivalence` | "Product of USA" | "United States" | `approve` |
| `country-wrong-import` | "Made in Canada" | "USA" | `review` (LLM resolver punts to human) |
| `class-whisky-whiskey` | "Kentucky Whisky" | "Kentucky Whiskey" | `approve` |
| `class-totally-different` | "Stout Beer" | "Vodka" | `review` |
| `appellation-without-vintage` | "2020 Napa Valley" | "Napa Valley" no year | `review` |

Each kind has a deterministic RNG seed so re-generating with the same
seed produces the same image + declared-fields pair. That gives us a
stable eval set that can grow without manual curation.

### Client side

- New toolbench button: **"Generate synthetic label (seeded)"** with
  a small dropdown to pick a defect kind (or "random").
- Same behavior as the live COLA fetch: populates image + form,
  reviewer hits VERIFY.
- After the report comes back, compare `report.verdict` vs the stored
  `expectedVerdict` and show a small badge in the toolbench:
  "✓ matched expected" or "✗ expected `approve`, got `review`".

### Eval harness integration

- Extend `scripts/evals/remote-eval.ts` with a `--slice=synthetic` option
  that iterates the manifest, calls `/api/eval/synthetic/generate`
  for each entry with a deterministic seed, and scores against the
  stored `expectedVerdict`.
- This gives us a second accuracy column next to the cola-cloud-all
  real-corpus column so we can track real vs seeded coverage
  separately.

## Risks

- **Image generation is slow (~5-15s per label)** depending on model
  tier. The batch eval might take 5+ minutes on 28 synthetic cases.
  Cache generated bytes keyed by `(defectKind, seed)` so replays are
  instant.
- **Gemini may refuse to generate labels with intentional defects**
  (safety filters, brand-IP concerns). Mitigation: the brand names
  the generator uses are non-real ("Mill River Cider", "Stone Creek
  Bourbon") — we seed them from a fake-name library, not from any
  real TTB COLA. Defects are phrased structurally ("make the ABV on
  the label read 14.5%") rather than as adversarial prompts.
- **Image realism drift** — if Gemini's output doesn't look
  photorealistic, the extraction pipeline's VLM might not treat it
  like a real label and accuracy degrades. Fix by prompt tuning or
  by filtering through a post-generation quality check.

## Follow-up candidates (not in v1)

- Wine-specific variants: vintage year mismatches, missing
  appellation, disallowed grape varietals.
- Hard-case adversarial labels: warnings hidden on side strips,
  warnings with unusual fonts, labels that use both "ABV" and
  "% Alc./Vol." wording.
- Image-quality variants: blurry, partial occlusion, weird angles.
  These exercise the `image-quality=low-confidence` path.
