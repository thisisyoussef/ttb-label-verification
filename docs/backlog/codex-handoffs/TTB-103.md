### Layout or styling constraints Codex must preserve

1. **Shell continuity.** Top identification region (product title, tagline, `Single | Batch` toggle) is identical to TTB-101 / TTB-102. `Nothing is stored…` privacy anchor must remain visible on every batch surface.
2. **Two surfaces only.** `batch-intake` and `batch-processing`. The dashboard is TTB-104's surface; `Open Dashboard →` is the only bridge into it (currently a placeholder alert).
3. **Matching review is inline.** Not a modal. Not a separate route. Groups in the fixed order listed above; `Matched` collapses by default.
4. **Count strip order locked** per the template above; never rearrange.
5. **Progress readout text-first.** Bounded progress bar reinforces; never the other way around. No percentage without its `{done} of {total}` count.
6. **Partial-failure framing.** Per-item errors never abort the run. Errored rows appear in the stream with inline `Retry this item`; they do not reduce `{done}` toward `{total}`.
7. **Terminal tone tokens.**
   - mixed → `border-primary bg-primary-container/20`
   - all-pass → `border-tertiary bg-tertiary-container/20` (quiet, not celebratory)
   - all-fail → `border-error bg-error-container/15` (serious, not alarmist)
8. **Cap is a product constant.** `BATCH_CAP = 50`. Server must enforce the same cap and return a `fileErrors[].reason === 'over-cap'` entry when exceeded.
9. **Theme tokens only in UI.** No raw hex in `src/client/**`. The one exception is `src/client/labelThumbnail.ts`, which embeds hex values inside SVG image content (not UI surfaces). Codex may keep or replace this helper when wiring real image previews; UI only depends on `previewUrl` being a renderable `src`.
10. **Accessibility.** `prefers-reduced-motion` is honored on progress bar and hover transforms. Focus ring via `:focus-visible`. Every status is reinforced by icon + label, never color alone. Matching review groups are labeled regions with `aria-label={group}: {count}`. Progress readout uses `aria-live="polite"`. Image preview overlay is `role="dialog" aria-modal="true"` with focus trap + Escape / backdrop-click dismissal.
11. **Privacy posture (frozen).** The UI does not surface `Batch ID`, `Start Time`, `Operator ID`, per-row audit IDs, or any language implying durable workflow storage, assignment tracking, or review history. These were explicit Stitch inventions dropped during normalization — if engineering needs them for logging, they must not appear in the UI.
12. **No per-row thumbnails in "dashboardy" surfaces.** Stream thumbnails are kept deliberately small (40×52) and are click-targets for the overlay, not visual features. The full-size image lives in the overlay.

## Required Codex engineering work

The existing `POST /api/review` contract supports single-label only. Batch introduces three new integration points.

### Shared contract additions

Add to `src/shared/contracts/review.ts` (or a sibling `batch.ts`) once `TTB-301` is unblocked:

```ts
// Request
export const batchLabelFileSchema = z.object({
  clientId: z.string(),      // UI-generated stable id per file for the whole session
  filename: z.string(),
  sizeBytes: z.number().int().nonnegative(),
  mimeType: z.string()
});

export const batchPreflightRequestSchema = z.object({
  batchClientId: z.string(), // ephemeral session id, client-generated
  images: z.array(batchLabelFileSchema).max(50),
  csv: z.object({
    filename: z.string(),
    sizeBytes: z.number().int().nonnegative()
  })
});

// Actual multipart: labels[] + csv (same field names as single-label POST /api/review),
// plus the JSON preflight body above.

// Matching preflight response
export const batchCsvRowSchema = z.object({
  id: z.string(),
  rowIndex: z.number().int().positive(),
  filenameHint: z.string(),
  brandName: z.string(),
  classType: z.string()
});

export const batchMatchedPairSchema = z.object({
  imageId: z.string(),
  row: batchCsvRowSchema,
  source: z.enum(['filename', 'order'])
});

export const batchAmbiguousSchema = z.object({
  imageId: z.string(),
  candidates: z.array(batchCsvRowSchema).min(2)
});

export const batchFileErrorSchema = z.object({
  filename: z.string(),
  reason: z.enum(['unsupported-type', 'oversized', 'duplicate', 'over-cap']),
  message: z.string()
});

export const batchPreflightResponseSchema = z.object({
  batchSessionId: z.string(),            // server-assigned; ephemeral
  csvHeaders: z.array(z.string()),       // actual header row from the uploaded CSV
  matching: z.object({
    matched: z.array(batchMatchedPairSchema),
    ambiguous: z.array(batchAmbiguousSchema),
    unmatchedImageIds: z.array(z.string()),
    unmatchedRowIds: z.array(z.string())
  }),
  csvError: z.object({ message: z.string() }).optional(),
  fileErrors: z.array(batchFileErrorSchema).default([])
});

// Start-run request (after UI resolves ambiguity/unmatched)
export const batchResolutionSchema = z.object({
  imageId: z.string(),
  // Exactly one of: chosen row, dropped, paired unmatched, etc.
  action: z.discriminatedUnion('kind', [
    z.object({ kind: 'matched', rowId: z.string() }),
    z.object({ kind: 'dropped' })
  ])
});

export const batchStartRequestSchema = z.object({
  batchSessionId: z.string(),
  resolutions: z.array(batchResolutionSchema)
});

// Stream frames
export const batchItemStatusSchema = z.enum(['pass', 'review', 'fail', 'error']);

export const batchStreamProgressSchema = z.object({
  type: z.literal('progress'),
  done: z.number().int().nonnegative(),
  total: z.number().int().positive(),
  secondsRemainingEstimate: z.number().int().nonnegative().optional()
});

export const batchStreamItemSchema = z.object({
  type: z.literal('item'),
  itemId: z.string(),
  imageId: z.string(),
  filename: z.string(),
  identity: z.string(),                 // "brandName — classType" verbatim
  status: batchItemStatusSchema,
  reportId: z.string().optional(),      // drill-in handle for TTB-104
  errorMessage: z.string().optional()   // present iff status === 'error'
});

export const batchStreamSummarySchema = z.object({
  type: z.literal('summary'),
  total: z.number().int().nonnegative(),
  pass: z.number().int().nonnegative(),
  review: z.number().int().nonnegative(),
  fail: z.number().int().nonnegative(),
  error: z.number().int().nonnegative(),
  dashboardHandle: z.object({ sessionId: z.string() })
});

export const batchStreamFrameSchema = z.discriminatedUnion('type', [
  batchStreamProgressSchema,
  batchStreamItemSchema,
  batchStreamSummarySchema
]);
```

### API / server behavior

1. `POST /api/batch/preflight` (multipart) — accepts `labels[]` + `csv` + a JSON `batchClientId`; returns `batchPreflightResponseSchema`. Do the CSV parse and filename-first / order-based matching here, so the UI can render the matching review before committing to a run.
2. `POST /api/batch/run` (JSON) — accepts `batchStartRequestSchema`; starts the processing pipeline for the resolved set; returns an SSE / chunked-JSON stream of `batchStreamFrameSchema` frames. UI prefers SSE, but any monotonic framing works.
3. `POST /api/batch/retry` (JSON) — accepts `{ batchSessionId, itemId }`; re-processes one errored item; replies with a single `batchStreamItemSchema` frame (or streams one back into the main run's channel — either is acceptable; UI currently assumes a fresh `reportId` on each retry).
4. `DELETE /api/batch/:batchSessionId` or `POST /api/batch/cancel` — cancels the run; server must stop spending OpenAI tokens promptly.

All endpoints honor `store: false` on every OpenAI call. No batch identity, file metadata, CSV content, or result payload is persisted beyond the live session.

### Validator / rule behavior

Per-item verification reuses the TTB-102 evidence model (`VerificationReport` from `TTB-201`). Batch adds no new rule families. The drill-in from the TTB-104 dashboard fetches a full `VerificationReport` by `reportId`; that fetch may be an in-memory lookup on the server (again, ephemeral).

### Loading, error, empty-state behavior the backend must support

- Matching preflight must distinguish `csvError` from `fileErrors[]`. A CSV parse failure returns `csvError` and no `matching`; individual file rejections return `fileErrors` entries and still attempt matching on the remaining images.
- Duplicate image filenames within a single batch return a `fileErrors[].reason === 'duplicate'` entry rather than being silently de-duplicated.
- Over-cap (>50 images) returns `fileErrors[].reason === 'over-cap'`.
- `unmatchedImageIds` / `unmatchedRowIds` are surfaced even when `ambiguous` is also present — UI renders the four groups independently.
- During a run, per-item errors stream as `status === 'error'` frames with an `errorMessage`; the run continues for remaining items.
- Cancellation mid-run is a client action (`POST /api/batch/cancel`). The stream ends cleanly; no further frames are expected.
- Empty terminal summary (total 0) is possible only if every item was dropped pre-run; UI tolerates it with the `No labels finished.` intent line.

### Privacy / no-persistence constraints

- `store: false` on every OpenAI call.
- No image bytes, CSV rows, matched identity strings, per-item results, or drill-in payloads may be persisted outside the live session.
- No logging of raw extracted text, brand names, filenames, or CSV contents in a way that persists.
- `batchSessionId` is session-scoped; a page reload discards it. UI does not attempt restore.
- `dashboardHandle.sessionId` is likewise ephemeral — the TTB-104 dashboard fetches results by it from an in-memory store that dies with the session.
- The UI deliberately surfaces no `Batch ID`, `Operator ID`, or timestamps beyond the progress readout's ETA. Server must not send those fields even if convenient for logging.

### Latency constraints

- Single-label 5-second budget stays unchanged; each item in a batch honors it independently.
- Matching preflight should complete in < 2 seconds for a 50-label / 50-row batch. UI shows no preflight spinner currently; if Codex expects longer, flag back to Claude for a progress affordance.
- First stream frame within 1 second of `POST /api/batch/run` acceptance.
- Stream cadence at least one frame every 500ms when items are completing; cadence is a UX concern — if real throughput is lumpier, consider synthesizing a periodic `progress` heartbeat.
- UI's seeded retry affordance assumes a new `reportId` per attempt; server should honor that.

### Tests to derive from the accepted behavior

- CSV parse fixtures covering: missing required headers, malformed rows, duplicate filename column values, empty CSV, CSV-only-with-BOM.
- Matching algorithm: filename-first with case-insensitive extension matching, order-based fallback when `filename` column is empty for a row, ambiguity detection when two rows share a normalized filename, zero-match images, zero-match rows.
- Over-cap rejection at 51 images with the exact `over-cap` reason string.
- Stream cadence tests: no frame lost under concurrent item completion, `progress` frames monotonically increase `done`, final `summary` frame arrives after all `item` frames.
- Cancellation: no further frames after cancel, no OpenAI tokens spent for uncompleted items after the cancel request is accepted.
- Privacy tests: no filename / CSV content / report payload appears in persistent storage or logs after a session ends.

## Open questions or blockers

1. **Stream transport.** SSE vs. chunked JSON vs. polling. UI prefers SSE; it only needs monotonic updates. No design blocker either way — flag if you pick polling so we can add a visible heartbeat.
2. **Seconds-remaining estimate.** Is the server expected to emit a per-frame ETA? If not, UI will fall back to a simple smoothed average derived from `progress` frames on the client.
3. **Matching by content vs. filename.** Current UI assumes filename-first matching; if Codex wants to add image-perceptual matching later, that changes the "matched by filename / row order" copy and needs a new UI pass.
4. **Session restore on reload.** Out of scope for the POC. If that becomes desired, UI needs a session-restore path and the `batchSessionId` needs a TTL.
5. **Dashboard handle stability.** Does `dashboardHandle.sessionId` match the `batchSessionId`, or is it a separate handle? UI doesn't care functionally, but TTB-104 will want to know.
6. **PDF labels.** UI currently renders PDF-flagged labels with an icon in the stream and a dedicated "PDF preview unavailable" state in the overlay. If Codex wants to render PDF first-page previews, that's a capability addition, not a UI redesign.
7. **Retry framing.** On retry of an errored item, does the server replace the errored frame in-stream or append a new frame? UI currently appends with a fresh `retryKey`. Either works.
