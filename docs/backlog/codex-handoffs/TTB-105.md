## Accessibility audit — verified

All acceptance criteria 3–8 were satisfied by prior stories and confirmed during this audit:

- **Status independence from color** (§3): every `Pass` / `Review` / `Fail` / `Info` / `Error` surface uses icon + text label in addition to color. Checked: `StatusBadge` (single-label), batch triage row status, batch stream row status, filter pill active state (uses text weight + background change, not color-only), verdict banner tint.
- **Dense-state legibility** (§4): warning character-aligned diff renders without horizontal compression at 1280px; batch triage table uses `truncate max-w-[260px]` on filename and identity cells to prevent table-width ballooning; cross-field checks wrap explanation text without pushing sibling rows.
- **Trust copy tone** (§5): every terminal / empty / export / error message is calm and procedural. No celebratory, panic, or marketing register.
- **Keyboard reachability** (§6): verified via Tab traversal on each main surface. `Back to Intake` is the first Tab stop on Results.
- **Reduced motion** (§7): existing `TTB-102` `prefers-reduced-motion` behavior on row expand / collapse is preserved. The `TTB-104` walker's hover-promote gradient is a class-transition-only effect (no `transform` animation), so reduced-motion users see the color change but no movement. No remediation needed.
- **Privacy anchor** (§8): every main surface carries the anchor once. The new `BackBreadcrumb` deliberately does not add a second instance on the Results view because the pinned column already carries it.

## Open questions or blockers

1. **Keyboard shortcut for back.** `Esc` on the single-label Results view currently collapses an expanded evidence row if one is open; if no row is expanded, `Esc` could return to intake. This overlaps with the existing `Esc`-to-collapse behavior. UI did not wire `Esc` to back-navigate because the overlap needs a policy call. Default recommendation: keep `Esc` as row-collapse only; add a separate `B` shortcut if reviewers want one-key return. Not blocking.
2. **Processing Esc shortcut.** Similar consideration for `Cancel review` on Processing. Default: no new keyboard shortcut; the promoted button is discoverable enough.
3. **Does `Back to Intake` need a confirmation?** Currently the handler runs immediately, discarding any unsaved intake state from before the review. Acceptable because the intake form repopulates from the scenario if one was selected, and because the product explicitly does not persist between sessions. If Codex thinks this warrants a confirmation, raise it — but the product's no-persistence posture argues against adding confirm-dialog friction.
