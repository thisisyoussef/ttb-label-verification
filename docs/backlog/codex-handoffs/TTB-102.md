## Frozen design constraints Codex must preserve

These are fixed inputs. Codex does not change them without a new Claude pass.

1. **Shell continuity.** Top identification region and `Single | Batch` toggle are identical to `TTB-101`. Pinned left image column (thumbnail, filename, size, beverage-type pill) is identical in structure and position across processing and results. Privacy assurance `Nothing is stored. Inputs and results are discarded when you leave.` is anchored at the bottom of the pinned column on results.
2. **Results layout.** Two columns at desktop (≥`md`: 4/8 split, ≥`lg`: 3/9). On narrower viewports the image column stacks above the working area. Max working-area content width is constrained by the shell's 1400px header wrapper.
3. **Checklist-first hierarchy.** `Results` heading → verdict banner (with inline counts) → (standalone banner when applicable) → field checklist → cross-field section → action bar. Never reorder.
4. **Exactly one row expanded at a time.** Opening a new row auto-collapses the previous one. Row uses `aria-expanded` + `aria-controls` + chevron rotation.
5. **Warning row uses a distinct expanded panel.** Sub-checks + confidence on a two-column grid, followed by a full-width character-aligned diff and legend. Non-warning rows use the standard evidence panel (summary + explanation + severity + confidence + citations + optional comparison block).
6. **Status vocabulary exact:** `Pass`, `Review`, `Fail`, `Info`. Not `PASSED`, `FORMAT PASS`, etc. Severity vocabulary exact: `Blocker`, `Major`, `Minor`, `Note` (shown in expanded panel only).
7. **Verdict headlines exact:** `Recommend approval`, `Recommend manual review`, `Recommend rejection`. Low-confidence secondary is `Low extraction confidence — review carefully.` Blocker secondary uses the template `{Check label} is the deciding check.`
8. **Standalone mode is an info band + column-level change.** Not a separate page or route. Application-value column is removed from checklist rows when `standalone === true`. Cross-field checks that depend on application data render with `status === 'info'` and a note or are omitted; UI tolerates both.
9. **No-text is a first-class state.** Replaces the verdict region entirely with a neutral advisory band and two actions (`Try another image` / `Continue with caution`). Checklist and cross-field section are hidden unless the reviewer chooses `Continue with caution`.
10. **Action bar copy exact:** `New Review` (primary), `Export Results` (secondary), `Run Full Comparison` (added before `New Review` in standalone mode). Keyboard shortcut `N` triggers New Review anywhere outside a form field. Tooltip on the primary button reads `Press N to start a new review.`
11. **Reviewer-action buttons do not live inside evidence panels.** No `IGNORE` / `APPROVE MATCH` / `CONFIRM REJECTION` buttons inside expanded rows. Evidence panels are read-only.
12. **Theme tokens only.** No raw hex in `src/client/**`. All status colors flow through `tertiary` / `caution` / `error` / `secondary` families plus their containers. The caution palette is the only TTB-102 theme addition; extend the palette in `tailwind.config.js` and `docs/design/INDUSTRIAL_PRECISION_THEME.md` together, not separately.
13. **Accessibility.** `prefers-reduced-motion` is honored. Focus ring via `:focus-visible`. Every verdict/status/severity is reinforced by icon + label, never color alone. Warning diff segments carry per-segment `aria-label` and a live-region summary. Keyboard paths: Tab / ArrowUp / ArrowDown / Enter / Escape / `N`.
14. **Privacy.** UI does not log result values (extracted values, confidence numbers, citations) to console / storage / analytics. Image `previewUrl` is revoked on any path that clears the image (`New Review`, `Try another image`, existing `Clear`).

### Response shape additions

Expand `VerificationReport` (shared contract owned by Codex in `TTB-201`) with these additions:

```ts
// src/shared/contracts/review.ts
export const verdictSchema = z.enum(['approve', 'review', 'reject']);
export const checkStatusSchema = z.enum(['pass', 'review', 'fail', 'info']);
export const severitySchema = z.enum(['blocker', 'major', 'minor', 'note']);

export const extractionQualityStateSchema = z.enum([
  'ok',
  'low-confidence',
  'no-text-extracted'
]);

export const extractionQualitySchema = z.object({
  globalConfidence: z.number().min(0).max(1),
  state: extractionQualityStateSchema,
  note: z.string().optional()
});

export const verificationCountsSchema = z.object({
  pass: z.number().int().nonnegative(),
  review: z.number().int().nonnegative(),
  fail: z.number().int().nonnegative()
});

export const comparisonStatusSchema = z.enum([
  'match',
  'case-mismatch',
  'value-mismatch',
  'not-applicable'
]);

export const comparisonEvidenceSchema = z.object({
  status: comparisonStatusSchema,
  applicationValue: z.string().optional(),
  extractedValue: z.string().optional(),
  note: z.string().optional()
});

export const warningSubCheckIdSchema = z.enum([
  'present',
  'exact-text',
  'uppercase-bold-heading',
  'continuous-paragraph',
  'legibility'
]);

export const warningSubCheckSchema = z.object({
  id: warningSubCheckIdSchema,
  label: z.string(),
  status: checkStatusSchema,
  reason: z.string()
});

export const diffSegmentKindSchema = z.enum([
  'match',
  'missing',
  'wrong-character',
  'wrong-case'
]);

export const diffSegmentSchema = z.object({
  kind: diffSegmentKindSchema,
  required: z.string(),
  extracted: z.string()
});

export const warningEvidenceSchema = z.object({
  subChecks: z.array(warningSubCheckSchema).length(5),
  required: z.string(),
  extracted: z.string(),
  segments: z.array(diffSegmentSchema)
});

export const fieldReviewSchema = z.object({
  id: z.string(),
  label: z.string(),
  status: checkStatusSchema,
  severity: severitySchema,
  summary: z.string(),
  details: z.string(),
  confidence: z.number().min(0).max(1),
  citations: z.array(z.string()).default([]),
  applicationValue: z.string().optional(),
  extractedValue: z.string().optional(),
  comparison: comparisonEvidenceSchema.optional(),
  warning: warningEvidenceSchema.optional()
});

export const verificationReportSchema = z.object({
  id: z.string(),
  mode: z.enum(['single-label', 'batch']),
  beverageType: beverageTypeSchema,
  verdict: verdictSchema,
  verdictSecondary: z.string().optional(),
  standalone: z.boolean(),
  extractionQuality: extractionQualitySchema,
  counts: verificationCountsSchema,
  checks: z.array(fieldReviewSchema),
  crossFieldChecks: z.array(fieldReviewSchema),
  latencyBudgetMs: z.number().int().positive().max(5000),
  noPersistence: z.literal(true),
  summary: z.string()
});
```

This replaces the existing `overallStatus` / `recommendation` single-verdict model with a richer `verdict` + `counts` + `extractionQuality` + `crossFieldChecks` model. `TTB-201` should keep the existing seed report working during the migration (dual-write or migrate the seed at the same time); the UI already tolerates a `standalone: true` report with or without application values.

### Warning evidence requirements

On the `government-warning` row only, include `warning`:

- `subChecks` always contains all five canonical sub-check IDs in the canonical order (`present`, `exact-text`, `uppercase-bold-heading`, `continuous-paragraph`, `legibility`). Labels must match the brief §6 anchors verbatim. Reason strings are plain English, no codes, no timestamps.
- `required` is the canonical warning text verbatim.
- `extracted` is what the system read off the label verbatim.
- `segments` is an ordered list walking through `required` left to right. Each segment has a kind and the corresponding chars from both strings. For `missing`, `extracted` is the empty string (or a single padding char). For `wrong-case` and `wrong-character`, both strings are non-empty and the same logical length.

The UI renders segments positionally. Positional alignment is the server's responsibility — the client does not re-compute the diff.

### Comparison evidence requirements

For non-warning rows that are comparisons:

- `comparison.status === 'match'` when application value and extracted value are equivalent after normalization.
- `comparison.status === 'case-mismatch'` when they differ only in letter case. UI highlights in `caution`.
- `comparison.status === 'value-mismatch'` when they differ beyond casing. UI highlights in `error`.
- `comparison.status === 'not-applicable'` when no comparison applies (e.g., standalone mode, or no application value provided for that field).
- `applicationValue` and `extractedValue` inside `comparison` are the exact strings to render in the comparison block. They may be omitted (UI falls back to the outer `applicationValue` / `extractedValue`).

### Standalone mode

The backend returns `standalone: true` when the reviewer provided no application data. The UI does not fabricate missing application values. Cross-field checks that require application data should:

- **Prefer:** return with `status: 'info'` and a `note` explaining the skip.
- **Acceptable fallback:** omit them. UI renders the empty cross-field state with the "No cross-field checks apply to this label." copy in that case.

### No-text-extracted

`extractionQuality.state === 'no-text-extracted'` is a first-class API state. `counts` may be zero across the board; `checks` may be empty. UI renders the dedicated no-text state. If the reviewer chooses `Continue with caution`, the UI currently swaps the client state to the `low-quality-image` seeded report. If Codex prefers to serve a real partial report for that case, return a full `VerificationReport` with `extractionQuality.state === 'low-confidence'` and flag every row `review`.

### Error → results boundary

Errors surfaced during processing (network, timeout, adapter, validation) continue to use the `TTB-101` failure variant via `reviewErrorSchema`. The results view never receives a request-level error — it only receives `VerificationReport`.

### Export payload

The UI currently exports the in-memory `UIVerificationReport` to JSON via a client-side `Blob` download. Codex does not need to implement a server export route for this story. If Codex wants to gate export behind a feature flag (e.g., only enabled after the live report shape lands), it may; the UI will honor `exportEnabled={false}` by disabling the button and showing the deferred tooltip `Export is enabled when the live pipeline lands.`

## Supported UI states to preserve

Comparison mode:

- Approve (all pass): verdict banner tertiary tint, counts pills, clean cross-field empty state.
- Review (cosmetic comparison): verdict banner caution tint, one row expanded with comparison block showing `case-mismatch`.
- Reject (warning defect): verdict banner error tint, warning row expanded with sub-check failures, confidence panel, and character-aligned diff.
- Reject (cross-field dependency): verdict banner error tint, cross-field row expanded with summary/explanation/citations.
- Low confidence global: verdict `review`, secondary copy `Low extraction confidence — review carefully.`, affected rows show amber/red confidence meters.

Standalone mode:

- Info band visible between counts and checklist.
- Application-value column absent.
- Cross-field section shows the skip explanation or the empty-state copy.
- Action bar includes `Run Full Comparison` before `New Review`.
- `Run Full Comparison` returns to intake with extracted values pre-filled into `brandName`, `fancifulName`, `classType`, `alcoholContent`, `netContents`, `applicantAddress` (when present in the report).

No-text-extracted:

- Neutral advisory band replaces verdict; checklist hidden.
- `Try another image` clears the image only, preserves form fields, returns to intake.
- `Continue with caution` stays on results and swaps to a low-confidence report.

## Privacy constraints Codex must preserve

- `store: false` on every OpenAI call (unchanged from `TTB-101`).
- Server must never log raw extracted text, diff payloads, sub-check reasons, application field values, or citations in a way that persists.
- Client must not log result values to console / `localStorage` / `sessionStorage` / analytics. Currently respected — do not introduce telemetry without a privacy review.
- Export blob URLs are revoked immediately after the download triggers.
- Image `previewUrl` is revoked on `New Review`, `Try another image`, and `Clear`.

## Performance constraints Codex must preserve

- Sub-5-second single-label end-to-end target unchanged.
- Results first paint target: under 150ms on a mid-tier laptop after `phase === 'terminal'`.
- Row expansion animation ≤300ms, instant under `prefers-reduced-motion`.
- Warning diff must remain interactive with up to ~1,000 canonical characters without virtualization.

## Open engineering questions

1. **Contract cutover.** Will `TTB-201` land the expanded `verificationReportSchema` above as a breaking change (updating the existing seed simultaneously), or add a parallel schema and phase out the old one? UI can run against either path as long as the shape above is eventually reachable.
2. **Diff segment computation.** Server-side alignment logic — any shared library preference, or greenfield? The UI's rendering assumes positional alignment is correct; a buggy aligner will produce nonsense highlights.
3. **Run Full Comparison payload.** Current UI behavior copies the extracted values client-side into the intake form. Does the server need a separate `POST /api/review/prefill` or similar to return a canonical prefilled form? Default is "no, keep it client-side."
4. **Export wiring.** Keep the client-side JSON export as-is, or land a server export route? If server-owned, what header shape (CSV? PDF?) and does it bypass `store: false`? Default: keep client-side for the proof of concept.
5. **Keyboard `N` shortcut.** Should this be gated behind a `data-keyboard-shortcuts-enabled` attribute in case Codex introduces conflicting global shortcuts?
6. **Empty checks array.** When `extractionQuality.state === 'no-text-extracted'`, the UI does not render the checklist. If `counts` are present (e.g., all zeros), that's fine. Does Codex want the server to always return empty `checks` in that case, or return a partial best-effort checks array that the UI simply hides until `Continue with caution`?
