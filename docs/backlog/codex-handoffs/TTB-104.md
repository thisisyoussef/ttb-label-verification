### Interaction states approved

Dashboard (seed IDs in `batchDashboardScenarios.ts`):

1. `dash-terminal-mixed` — six-label batch, realistic spread of approve/review/reject; default filter `All`; default sort `Worst first`.
2. `dash-with-error` — five completed rows plus one `Error` row with `Retry this item`. Clicking retry transitions the row to `Review` with a new report and updates the counts + summary cards.
3. `dash-terminal-all-pass` — six approves; quieter tertiary-tinted tone.
4. `dash-terminal-all-fail` — six rejects; serious error-tinted tone.
5. `dash-cancelled-partial` — three completed rows of twelve started; secondary intent line `3 reviewed of 12 started · Batch cancelled`.

Filter states: `All` / `Rejects only` / `Reviews only` / `Approves only`. Filter empty renders `No {filter} in this batch.` with `Clear filter` action (e.g., `Approves only` on `dash-terminal-all-fail`).

Sort states: `Worst first` (default), `Filename`, `Brand name`, `Completed order`.

Export states: idle → confirming → in-progress → idle (on success) or error → retry. Export confirmation is inline in the action bar, never a modal.

Drill-in shell:

6. `Available` — wraps the approved `Results` component, which renders its own verdict banner + checklist + cross-field + evidence panels using the underlying `VerificationReport`. The shell adds a top orientation nav (Back to Batch Results + position chip) and a bottom sticky walker (Previous label · position · Next label). Previous/Next live in the walker only — they are deliberately absent from the top nav so the primary adjacent-label action is discovered where the eye naturally lands after reading evidence.
7. `First in view` — `Previous label` in the walker is visually inert (muted background, no hover promote).
8. `Last in view` — `Next label` in the walker is visually inert.
9. `Unavailable` — advisory panel `This label's details aren't available. It may have errored during processing.` with only `Back to Batch Results`. Walker is not rendered for unavailable rows.

"Reviewed this session" row indicator:

10. On every drill-in open, the row's id is added to a session-only Set. Rows in that Set show a small `Reviewed this session` marker under the filename. The Set clears on `Start Another Batch`, on dashboard-seed change (dev), and on page reload.

- `low-quality-image` → dashboard row Review with `review · low confidence` in the Issues cell → drill-in shows the low-confidence banner secondary + amber/red confidence meters.

### Layout or styling constraints Codex must preserve

1. **Shell continuity.** Top identification region (product title, tagline, `Single | Batch` toggle) and `Nothing is stored…` privacy anchor are identical to TTB-101 / TTB-102 / TTB-103. No sidebar, no user profile surface, no "Compliance Hub / History / Standards" nav, no footer nav links.
2. **Three summary cards, fixed order.** `Approve`, `Review`, `Reject`. `Error` is not a fourth card — it appears only in row status and is implicit in the `All` filter count.
3. **Filter pills, fixed order.** `All`, `Rejects only`, `Reviews only`, `Approves only`. Each renders `{Label} · {count}` with the count reflecting the full-batch totals (not post-filter), so the reviewer always knows what they're filtering out of.
4. **Sort control, fixed options and default.** `Worst first` is the default. The other three options are offered in the exact order in the brief.
5. **Triage table columns, fixed count and order.** Five columns: `Status`, `Label`, `Identity`, `Issues`, `Actions`. Do not re-split `Identity` into separate brand / class columns — the composed identity cell (brand on line 1, class + beverage type on line 2) is the approved composition.
6. **Worst-first sort semantics.** Status rank `fail < review < pass < error`; within a status group, higher blocker/major/minor/note weights rise; ties break on `completedOrder`. Error rows sink to the bottom of the `All` view and are never shown under `Rejects only` / `Reviews only` / `Approves only` filters.
7. **Drill-in is a reuse of the TTB-102 Results view verbatim.** The shell does not modify the Results region. The shell's only added UI is the top orientation nav (Back to Batch Results + position chip), the sticky bottom walker (Previous label · position · Next label), and the detail-unavailable panel (which replaces Results entirely when `reportId === null`; walker is not rendered in that case).
7a. **Drill-in shell spans full viewport width.** Top nav and walker use `w-full px-6 lg:px-8` — no `max-w` centering — so they align edge-to-edge with the full-viewport Results grid below. Keeping them centered at 1400px leaves the `Back` button floating ~260px inset from the viewport edge on wide monitors, which reads as centered-in-empty-space. The app's main header still uses its existing `max-w-[1400px] mx-auto` pattern and is not changed by this story.
7b. **Walker is sticky.** `sticky bottom-0 z-30` with upward shadow + `backdrop-blur` so Prev / position / Next stay visible during long-evidence scroll. When scrolled to the absolute bottom of the page, the walker lands at its natural position after the Results action bar rather than continuing to hover — preventing permanent overlap of `New Review` / `Export Results`.
7c. **Previous/Next are not in the top nav.** Only in the bottom walker. Duplicating them in both places created a discoverability miss in user testing.
8. **Export is inline, not modal.** Export confirmation / in-progress / error all render in place of the `Export Results` button inside the action bar.
9. **Export output is one JSON download.** Not a preview pane, not a multi-step wizard, not multiple files. CSV / PDF export is out of scope for this story.
10. **"Reviewed this session" is session-local.** Never persisted. Indicator disappears on `Start Another Batch`, on page reload, or when the dashboard seed changes (dev picker).
11. **Privacy anchor on every surface.** Dashboard, drill-in, export-confirmation, export-in-progress, export-error all carry `Nothing is stored…` language.
12. **`Open Dashboard →` from TTB-103 routes here.** The placeholder `window.alert` in `BatchProcessing.tsx` is replaced with a view transition to `batch-dashboard`; no other code in `BatchProcessing.tsx` changes.
13. **Theme tokens only in UI.** No raw hex in `src/client/**`. The `labelThumbnail.ts` SVG-content hex exception from TTB-103 continues to apply for dashboard row thumbnails.
14. **Accessibility.** Filter strip is `role="tablist"` with `aria-selected` on the active pill. Triage table is a `<table>` with semantic headers. Export confirmation uses `aria-live="polite"` during the in-progress state. Drill-in breadcrumb has `aria-label="Back to batch results"`. Every status is icon + text + color — never color alone. `prefers-reduced-motion` is honored on hover transforms and the export spinner.
15. **No analytics-dashboard energy.** No trend charts, no sparklines, no pie/donut, no "batch quality score", no time-series. No assignment column. No "last modified" timestamps. No audit IDs on rows. These were explicit Stitch inventions dropped during normalization.

### Shared contract additions

Add to `src/shared/contracts/review.ts` (or a sibling `batch.ts`) under `TTB-301`:

```ts
// Dashboard payload
export const batchDashboardIssuesSchema = z.object({
  blocker: z.number().int().nonnegative(),
  major: z.number().int().nonnegative(),
  minor: z.number().int().nonnegative(),
  note: z.number().int().nonnegative()
});

export const batchItemStatusSchema = z.enum(['pass', 'review', 'fail', 'error']);

export const batchDashboardRowSchema = z.object({
  rowId: z.string(),
  reportId: z.string().nullable(),
  imageId: z.string(),
  filename: z.string(),
  brandName: z.string(),
  classType: z.string(),
  beverageType: beverageTypeSchema,
  status: batchItemStatusSchema,
  previewUrl: z.string().nullable(),
  isPdf: z.boolean(),
  sizeLabel: z.string(),
  issues: batchDashboardIssuesSchema,
  confidenceState: extractionQualityStateSchema,
  errorMessage: z.string().nullable(),
  completedOrder: z.number().int().positive()
});

export const batchDashboardSummarySchema = z.object({
  pass: z.number().int().nonnegative(),
  review: z.number().int().nonnegative(),
  fail: z.number().int().nonnegative(),
  error: z.number().int().nonnegative()
});

export const batchDashboardResponseSchema = z.object({
  batchSessionId: z.string(),
  phase: z.enum(['complete', 'cancelled-partial']),
  totals: z.object({
    started: z.number().int().nonnegative(),
    done: z.number().int().nonnegative()
  }),
  summary: batchDashboardSummarySchema,
  rows: z.array(batchDashboardRowSchema)
});

// Drill-in: reuses existing VerificationReport shape
// Export: full session JSON with `rows` + `reports` + `noPersistence: true`
export const batchExportPayloadSchema = z.object({
  generatedAt: z.string().datetime(),
  phase: z.enum(['complete', 'cancelled-partial']),
  totals: z.object({
    started: z.number().int().nonnegative(),
    done: z.number().int().nonnegative()
  }),
  summary: batchDashboardSummarySchema,
  rows: z.array(batchDashboardRowSchema),
  reports: z.record(z.string(), verificationReportSchema),
  noPersistence: z.literal(true)
});
```

### API / server behavior

1. `GET /api/batch/:batchSessionId/summary` — returns `batchDashboardResponseSchema`. Called when the UI enters `batch-dashboard`.
2. `GET /api/batch/:batchSessionId/report/:reportId` — returns a full `VerificationReport`. Called when the UI enters `batch-result`. The UI passes this directly to the approved `Results` component.
3. `GET /api/batch/:batchSessionId/export` — returns `batchExportPayloadSchema` as a JSON file with `Content-Disposition: attachment; filename="ttb-batch-<sessionId>.json"`. Called on `Confirm export`. The UI may alternatively generate the same shape client-side from already-fetched data; server-side is preferred for consistency.
4. `POST /api/batch/:batchSessionId/retry/:imageId` — errored-row retry (reuses the TTB-103 handoff's retry endpoint). On success the row transitions to one of `pass` / `review` / `fail` and the dashboard payload is refreshed (either via a fresh summary fetch or by the server pushing an update).

All endpoints honor `store: false` on every OpenAI call. No batch session data is persisted beyond the live session.

### Validator / rule behavior

Drill-in reuses the TTB-102 evidence model unchanged. Dashboard introduces no new rule families. The `issues.blocker / major / minor / note` breakdown is computed from the underlying report's `checks` + `crossFieldChecks` where `status !== 'pass'` and `status !== 'info'`, grouped by `severity`. UI currently computes this client-side from the full report; if Codex prefers to precompute it server-side in the dashboard payload, the UI already expects the shape above.

### Loading, error, empty-state behavior the backend must support

- Dashboard summary 404 / 410 indicates an expired session; UI currently assumes session lives for the lifetime of the page, so this is an edge case.
- `GET /report/:reportId` 404 for a `reportId === null` row is expected; UI routes that case to the unavailable panel before making a fetch.
- Cancelled-partial dashboards must report `phase: 'cancelled-partial'` + `totals.done < totals.started`; UI renders the secondary intent line accordingly.
- Export errors return a JSON problem detail; UI shows `Export didn't complete. Try again.` with `Retry`.

### Privacy / no-persistence constraints

- `store: false` on every OpenAI call.
- No dashboard payload, report payload, or export artifact is persisted beyond the live session.
- Export blob URLs are revoked immediately after the download triggers (already implemented client-side).
- The reviewed-ids Set is session-local and never sent to the server.
- `batchSessionId` is ephemeral; a page reload loses it. UI does not attempt restore.
- The UI deliberately surfaces no row-level audit IDs, operator IDs, or timestamps. Server must not send them even if convenient.

### Latency constraints

- Dashboard first paint < 150 ms after `Open Dashboard →` on a mid-tier laptop (already met with client-side seed; live should hold a similar budget).
- Drill-in enter < 200 ms.
- Filter / sort transitions < 50 ms (in-memory client-side).
- Export generation: aim for < 2 seconds for a 50-label session. If slower, keep the inline `Preparing your export…` state visible — UI already supports it.

### Tests to derive from the accepted behavior

- Dashboard summary payload validation: `batchDashboardResponseSchema` rejects malformed rows and missing totals.
- Worst-first sort: explicit unit tests covering `{fail-blocker-2} > {fail-blocker-1} > {review-major-3} > {pass} > {error}`.
- Filter counts: each pill's count matches the filter result exactly; no status leakage.
- Export shape: every row in `rows` with a non-null `reportId` has a matching entry in `reports`.
- Retry: an errored row's retry produces a new `reportId` and updates the summary counts consistently.
- Privacy: no dashboard / report / export payload persists after session end; no logs contain brand names, filenames, or extracted text.

## Open questions or blockers

1. **Dashboard retry UX.** Current UI optimistically transitions an errored row to `review` on retry. Server should confirm with a real outcome; if the retry itself errors again, the UI should restore the row to `error` with a fresh `errorMessage`. Default: UI expects a final outcome and updates the row in place.
2. **Session TTL.** How long is the dashboard reachable after a run completes? UI assumes lifetime of the page; a shorter TTL requires a visible "session expiring" affordance that this story does not design.
3. **Export format scope.** JSON only for the POC. CSV / PDF variants require a new story and a new UI pass. Do not add them into the existing export action.
4. **Previous / Next in drill-in respects the dashboard filter.** UI default: yes. If Codex implements server-side cursor pagination for drill-in, the cursor should reflect the current filter selection.
5. **"Reviewed this session" scope.** UI keeps this session-local. If Codex thinks an "already reviewed" signal should survive a reload, that's a new UX (and a privacy conversation); default: keep as-is.
6. **Concurrent drill-ins in tabs.** UI assumes single-page, single-tab. If Codex wants to support multiple concurrent drill-ins, the dashboard state model needs to accommodate it.
7. **Dashboard entry from outside a terminal.** Right now the only entry is `Open Dashboard →` after `TTB-103` processing finishes. If Codex exposes a recent-session URL or similar, the UI needs a new entry affordance.
