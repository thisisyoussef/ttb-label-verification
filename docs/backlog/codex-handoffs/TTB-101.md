## Frozen design constraints Codex must preserve

These are fixed inputs. Codex does not change them without a new Claude pass.

1. **Shell:** compact title bar (`TTB Label Verification Assistant` + subtitle `AI-assisted compliance checking`) + Single/Batch mode toggle. The shell is identical on intake and processing.
2. **Intake layout:** two-column at desktop (`lg:grid-cols-12` with 5/7 split), stacks on narrow viewports.
3. **Application-data groups, in order:** beverage type ŌåÆ Identity ŌåÆ Alcohol and measure ŌåÆ Origin and applicant ŌåÆ (conditional) Wine details.
4. **Beverage type:** exactly four options, verbatim strings ŌĆö `Auto-detect`, `Distilled Spirits`, `Malt Beverage`, `Wine`. Rendered as a segmented control, not a dropdown.
5. **Origin:** segmented control with `Domestic` and `Imported`. Country field appears only when Imported is selected.
6. **ABV tag behavior:** `MANDATORY` for Distilled Spirits and Wine; `OPTIONAL` for Malt Beverage; hidden for Auto-detect.
7. **Verify Label:** disabled until an image is present; disabled tooltip copy is exactly `Add a label image to verify.`; primary button uses the `primary ŌåÆ primary-dim` gradient with shadow ambient.
8. **Privacy microcopy:** `Nothing is stored. Inputs and results are discarded when you leave.` is always visible on intake in the action bar.
9. **Drop zone states:** empty, drag-over, uploaded, unsupported, oversized. Error copy is exact (see ┬¦6 of the brief).
10. **Processing steps:** five canonical steps in this order, verbatim ŌĆö `Reading label image`, `Extracting structured fields`, `Detecting beverage type`, `Running deterministic checks`, `Preparing evidence`. Step IDs used in the client state machine are `reading-image`, `extracting-fields`, `detecting-beverage`, `running-checks`, `preparing-evidence`.
11. **Step states:** only one step is `active` at a time; progression is linear; terminal state leaves all five `done` and keeps the dashed "Results will render here" placeholder visible.
12. **Failure variant:** current running step flips to `failed`, subsequent steps remain `pending`, failure panel shows heading `We couldn't finish this review.`, plain-English body (no error codes, no codes, no timestamps, no jargon), and two actions `Try again` and `Back to intake`.
13. **Processing sidebar:** thumbnail, filename (monospace), size, beverage-type pill, and a `Cancel review` error-colored link anchored at the bottom.
14. **Theme:** use only tokens in `tailwind.config.js`. No raw hex values in any `src/client/**` file. Follow `docs/design/INDUSTRIAL_PRECISION_THEME.md` when extending the palette.
15. **Accessibility:** focus ring via `:focus-visible`, `prefers-reduced-motion` respected, radiogroups are real radiogroups, every status has an icon + label independent of color, drop zone is keyboard-operable.

## Required data fields, API behavior, validation behavior

### Request contract (new `POST /api/review` endpoint to design)

- `multipart/form-data`.
- Part `label`: single file, one of `image/jpeg`, `image/png`, `image/webp`, `application/pdf`, Ōēż 10 MB. Reject anything else with a structured validation error before any model call.
- Part `fields`: JSON payload matching the intake form shape in `src/client/types.ts` ŌĆö specifically:
  ```ts
  {
    beverageType: 'auto' | 'distilled-spirits' | 'malt-beverage' | 'wine',
    brandName: string, fancifulName: string, classType: string,
    alcoholContent: string, netContents: string,
    applicantAddress: string, origin: 'domestic' | 'imported', country: string,
    formulaId: string, appellation: string, vintage: string,
    varietals: Array<{ name: string, percentage: string }>
  }
  ```
  All values are strings (the form does no numeric coercion). Empty strings mean "not provided" ŌĆö treat them as missing, not as invalid.

### Response contract

- Must conform to `VerificationReport` in `src/shared/contracts/review.ts`. TTB-101 does not render the report; TTB-102 does. The processing screen needs only to know when the call is complete so it can hand off.
- `store: false` on every OpenAI call. No persistence.
- End-to-end latency target: sub-5 seconds per the product invariants.

### Progress / streaming

The processing step list is driven by a fixed cadence in the client today (`STEP_ADVANCE_MS = 900`). Codex has two options:

1. **Stay with the fixed cadence** and have the `POST /api/review` return the final `VerificationReport` when ready. The UI will still show the five-step animation; this is acceptable for TTB-101.
2. **Stream progress** via Server-Sent Events (or similar). If streamed, emit events keyed by the exact step IDs above (`reading-image`, `extracting-fields`, `detecting-beverage`, `running-checks`, `preparing-evidence`) with states `active | done | failed`, followed by a terminal event carrying the `VerificationReport`. The client can swap its fixed cadence for the real event stream without re-rendering the shell.

### Error shape

Return a discriminated union the UI can map to the failure panel without parsing HTTP codes:

```ts
{
  kind: 'validation' | 'timeout' | 'network' | 'adapter' | 'unknown',
  message: string,         // plain English, shown as-is; no error codes
  retryable: boolean
}
```

### Validation behavior for TTB-101

Nothing yet. All intake fields are optional at the UI level. Oversized / unsupported files are caught client-side in the drop zone, but the server must also reject them. Per-field format validation (ABV format, wine appellation dependency, etc.) belongs in TTB-205.

## Supported UI states to preserve

Intake:

- Drop zone: empty, drag-over, uploaded, unsupported-file, oversized-file.
- Verify: disabled (with tooltip), enabled.
- Beverage type: Auto-detect (default) and each of the three explicit types; Wine reveals the wine group and varietals table.
- Origin: Domestic (country hidden) and Imported (country revealed).
- Varietals table: empty row placeholder, multiple rows, running total with color flip at 100%.
- Clear: idle + inline confirmation.

Processing:

- Running: any in-progress step, with earlier steps `done` and later steps `pending`.
- Terminal: all five done + dashed "Results will render here" placeholder.
- Failed: current step flipped to `failed`, failure panel visible with `Try again` + `Back to intake`.
- Cancel: returns to intake with image and fields intact.

## Privacy constraints Codex must preserve

- `store: false` on every OpenAI call.
- No durable writes. Uploaded files must be processed in-memory or in a per-request tmpfile deleted before the response is sent.
- Server must never log raw label text, application fields, or OCR output in a way that persists (no info logs of user-provided values in production build).
- Client already avoids logging values, persisting to storage, or sending telemetry. Do not introduce analytics without a new privacy review.

## Performance constraints Codex must preserve

- Sub-5-second single-label target (see `docs/presearch/2026-04-13-foundation.md` ┬¦"Target latency budget").
- The processing animation's fixed cadence (~4.5s total) must not leave the user waiting after the real pipeline completes. If the API returns faster than the cadence, the UI should be updated to skip ahead rather than stall ŌĆö flag this to Claude if streaming stays out of scope.

## Open engineering questions

1. **Streaming transport.** Will `POST /api/review` stream SSE step events, or will it return a single `VerificationReport`? Claude's UI supports both; default is the latter until Codex confirms.
2. **Day-one file formats.** UI accepts JPEG/PNG/WEBP/PDF. Confirm the extraction adapter supports PDFs at the start; if not, hide `PDF` from the intake copy until supported.
3. **"Auto-detect" semantics.** Does the server expect `auto` as a first-class value, or should the UI pass the extractor's best guess once available? Current contract sends `auto` verbatim when the user hasn't chosen a type.
4. **Country list.** Is free-text acceptable for the `country` field, or should we move to ISO 3166? UI currently uses free text.
5. **Varietal total enforcement.** UI only warns when total ŌēĀ 100%. Should the server reject, downgrade to review, or accept? This maps to TTB-205 validator work but the decision affects the intake UX copy.
6. **Force-failure + scenario-picker dev controls.** Keep behind env flag or remove for production? Claude has no opinion; Codex owns that decision.
