### Auth state machine (client-local, memory-only)

```
signed-out ──click PIV──▶ piv-loading ──(1200ms)──▶ piv-success ──(700ms)──▶ signed-in
signed-out ──click SSO──▶ sso-form   ──click Back──▶ signed-out
sso-form   ──submit────▶ sso-loading ──(1200ms)──▶ sso-success ──(700ms)──▶ signed-in
signed-in  ──Sign out──▶ confirm inline ──Confirm──▶ signed-out (+ reset app state)
                                         ──Cancel / 4.5s──▶ signed-in
```

- The state lives in React state only. No `localStorage`, no `sessionStorage`, no cookies, no tokens, no server round-trip.
- A page reload always drops back to `signed-out` — this is intentional prototype-safety posture.
- The SSO `User ID` input value is preserved if the reviewer clicks `← Back` from `sso-form` to `signed-out` and then re-enters SSO, for the duration of the tab session; a full reload drops it along with the rest of the state.

## Required Codex engineering work

1. **Regression tests** covering every transition in the state machine:
   - Initial load → `signed-out`.
   - PIV path: click PIV → `piv-loading` → (after ~1200ms) `piv-success` → (after ~700ms) `signed-in`.
   - SSO path: click SSO → `sso-form`; `Back` returns to `signed-out`; submitting any value → `sso-loading` → `sso-success` → `signed-in`.
   - Sign-out: `Sign out` → inline confirmation; Cancel returns to idle; Confirm returns to `signed-out` and clears app state.
   - Sign-out reset invariant: after `onSignOut`, `image === null`, `fields === emptyIntake()`, `beverage === 'auto'`, `scenarioId === 'blank'`, `variantOverride === 'auto'`, `report === null`, `mode === 'single'`, `view === 'intake'`, `batchPhase === 'intake'`, `drillInRowId === null`, `drillInRows === []`, `reviewedRowIds` empty, `exportState === { kind: 'idle' }`, `previewImage === null`, `tourOpen === false`, `tourStepIndex === 0`.
2. **No-persistence invariant** — no key starting with `ttb-auth`, `treasury-sso`, or similar is written to `localStorage` / `sessionStorage` / cookies by any auth path. (The existing `ttb-help:replay-state` from TTB-106 is unrelated and unaffected.)
3. **Timing-knob review** — `AUTH_TIMINGS.loadingMs` and `successMs` are exported from `authState.ts`. Codex may tune these (shorten under reduced-motion, for example), but the reduced-motion handling is currently done by dropping the spinner pulse animation; if Codex wants the transitions to also compress under reduced-motion, guard the `setTimeout` with a media-query check.
4. **No OpenAI, no fetch, no server route.** The story introduces zero backend surface. Do not add `/api/auth/*`, do not add cookie middleware, do not add any session infrastructure.

### Privacy / no-persistence constraints

- No tokens of any kind. No cookies. No server session.
- No reviewer identity logging (client or server).
- A page reload drops the auth state; Screen 0 is always the first paint on reload.
- The SSO `User ID` input value is never sent anywhere — it lives in component state and is discarded when the reviewer completes or abandons the flow.

### Latency constraints

- Screen 0 first paint < 100ms (no data fetch).
- PIV / SSO transitions run on `setTimeout` at the timings listed above; they are deliberately human-scale so the screen reads as institutional rather than instant. Under `prefers-reduced-motion` the spinner pulse is disabled; Codex may optionally compress the timings as well.

## Open questions or blockers

1. **Timing under reduced-motion.** Current behavior: pulse animation is disabled, but the `setTimeout`-driven transitions still run at 1200 + 700ms. Should the transitions also compress (e.g., to 150ms each) for reduced-motion users? UI default: leave them at full length so the state changes are still observable. Codex's call.
2. **Sign-out auto-dismiss window.** The inline confirmation auto-dismisses to Cancel after 4.5s. If Codex thinks that's brittle (reviewer reads it and doesn't act in time), remove the auto-dismiss and require an explicit click. UI default: keep the auto-dismiss so a wandering click elsewhere doesn't strand the confirmation.
3. **SSO User ID preservation on Back.** Currently preserved for the tab session — if the reviewer clicks Back and re-enters SSO, the typed value is still there. Codex may clear it if a fresh-form-is-cleaner argument emerges. UI default: preserve.
4. **Future PIV / SSO branching.** If the product ever needs a third sign-in method (e.g., `Sign in with Login.gov`), it drops into the same `CardBody` pattern with a new `AuthPhase` entry. Flagged for scope awareness only; no work for this story.
5. **`TTB-106` launcher placement after `TTB-107`.** With this story landed, the `Guided review` launcher now lives inside the signed-in shell's right cluster, which is exactly where TTB-106's handoff said it should be. No follow-up action required — the tour launcher is only reachable after sign-in, which is the correct scope.
