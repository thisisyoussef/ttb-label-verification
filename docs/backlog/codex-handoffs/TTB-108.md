## Hard constraints / non-negotiables

- **Local is the default.** Every sign-in and sign-out resets to `local`. This is deliberate — it signals deployment realism for Marcus and matches TTB's restricted-network posture.
- **Mode is tab-scoped React state.** No persistence (cookie, localStorage, or server-side). Cleared on sign-out.
- **The mode-select auth phase is not skippable.** Every sign-in passes through it.
- **Provider names never appear in the reviewer surface.** No "Gemini", "OpenAI", "Ollama", or "Qwen" in any user-facing copy.
- **Mode cannot change during processing.** The header switch link is hidden while a review is in flight.
- **Tooltip copy is grounded in real constraints.** The Local tooltip references the previous vendor pilot failure. Do not replace with generic descriptions.

## Flexible areas Codex may adjust

- The exact request-body shape for passing extraction mode to `POST /api/review`.
- The error kind and message for local-mode-unavailable (currently detected client-side as `extractionMode === 'local' && phase === 'failed'`; Codex should use a specific `ReviewError.kind` value).
- Copy refinements in the processing screen if the actual local extraction behavior differs from the placeholder descriptions.
- The header indicator styling if it conflicts with other header elements added in later stories.

## Relevant eval scenarios

- All six core-six scenarios work in both cloud and local mode (same seeded fixture data).
- The local-unavailable state can be triggered with "Force failure" fixture while in local mode.

## Required data fields and API behavior

- `POST /api/review` should accept an `extractionMode: 'cloud' | 'local'` field in the request body.
- When `extractionMode` is `local` and local extraction is not configured, respond with a `ReviewError` of `kind: 'adapter'` and a message the UI can display.
- When `extractionMode` is `cloud`, use the existing cloud extraction path.

## Supported states Codex must preserve

- Auth flow: signed-out → PIV/SSO loading → success → **mode-select** → signed-in
- Mode-select: Local pre-selected, Cloud available, Continue button advances to signed-in
- Header: mode indicator visible, switch link visible except during processing
- Processing: mode-aware header, body copy, and sidebar label
- Local unavailable: caution alert with Switch to Cloud and Back to intake actions

## Privacy and latency constraints

- No extraction mode choice is persisted anywhere.
- Local mode must not silently fall back to cloud. If local is selected and unavailable, the error must be explicit.
- Cloud mode continues to target the existing `<= 5,000 ms` budget.
- Local mode budget planning was moved into the archived `docs/specs/archive/TTB-212/` packet and is not part of the active implementation queue.

## Open questions / known engineering gaps

- The local-unavailable detection is currently a client-side heuristic (`extractionMode === 'local' && phase === 'failed'`). Codex should replace this with a proper error-kind check from the backend response.
- The actual `extractionMode` request field and backend routing land in `TTB-206`. This handoff provides the UI shell; `TTB-206` provides the backend routing.
