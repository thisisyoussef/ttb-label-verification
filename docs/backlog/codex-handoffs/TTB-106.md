### Interaction states approved

The 8-step tour, with targets and interactions:

1. **Orientation** — centered callout, no target. "A 2-minute tour." Click Next.
2. **Launcher** — spotlight on `tour-launcher`. "This button opens the tour any time."
3. **Drop a label image** — spotlight on `tour-drop-zone`. `Load sample` action loads `perfect-spirit-label` scenario into form state.
4. **Verify the label** — spotlight on `tour-verify-button` with `interaction: 'click-to-advance'`. CTA reads `Click Verify Label to continue` with a pulsing hand icon. A user click on the real button both runs verification and advances the tour.
5. **One verdict, backed by evidence** — spotlight on `tour-verdict-banner`. Target only renders on the Results view; falls back to a centered callout if the reviewer hasn't verified yet.
6. **Warning evidence** — spotlight on `tour-warning-row`. `Show warning defect` action loads `spirit-warning-errors` scenario.
7. **Switch to Batch** — spotlight on `tour-batch-tab` with `interaction: 'click-to-advance'`. CTA reads `Click the Batch tab to continue`. Clicking the tab both switches mode and advances the tour.
8. **Nothing is stored** — centered callout, no target. `Finish` button completes the tour and persists `tourCompleted: true`.

Info popovers at five semantic anchors (all click-to-open, Escape and outside-click to dismiss):

- `warning-evidence` — inside `WarningEvidence.tsx` next to the `Sub-checks` heading.
- `confidence-indicator` — inside `ConfidenceMeter.tsx` next to the `CONFIDENCE` label.
- `standalone-mode` — inside `StandaloneBanner.tsx`.
- `batch-matching-logic` — inside `MatchingReview.tsx` on the matching-rule explanation.
- `no-persistence` — inside `ResultsPinnedColumn.tsx` next to the privacy anchor copy (placement `bottom` because it sits at the bottom of the pinned column).

Replay-state behavior:

- **First load (unseen)** — launcher visible with a pointer-chip nudge `New here? Take a 2-minute tour.` below it. Chip dismisses permanently on `Dismiss` or on first tour launch.
- **Mid-tour** — launcher shows an active-state treatment (`bg-primary-container text-on-primary-container border-primary`) so the reviewer knows the panel is open.
- **Post-tour** — `tourCompleted: true` written to localStorage. No nudge appears on subsequent sessions. Re-launching from the button always starts at step 1 (tours are short — no mid-tour resume).

### Layout or styling constraints Codex must preserve

1. **Tour is a non-blocking overlay.** `pointer-events-none` on the outer container so the user can still interact with the real UI (required for click-to-advance). Only the callout card has `pointer-events-auto` so the Previous / Next / Finish / Close / Show me buttons work.
2. **Cutout is CSS `clip-path` polygon.** Not a modal backdrop. Not a portal. Not a second stylesheet. The dim color (`rgba(45, 52, 51, 0.55)`) derives from the `on-surface` token.
3. **Highlight ring uses existing tokens.** `border-primary` + a primary-tinted halo via `shadow-[0_0_0_4px_rgba(84,96,103,0.35)]`. No new tokens introduced.
4. **Callout positioning prefers below → above → right → left → center.** A 260px estimated callout height drives the space calculation; the final card renders at its natural height and is constrained to `min(420px, 92vw)` wide.
5. **Re-measure every 250ms.** Deliberate: scenarios load async, layouts shift on view changes, and the poll keeps the spotlight aligned without requiring explicit re-triggers. Codex may switch to `ResizeObserver` + `MutationObserver` later if the poll is too chatty.
6. **Click-to-advance uses capture-phase listener on `document`.** The tour catches the click before React state updates so the real element's own `onClick` still fires (e.g., the Batch tab both switches mode and advances the tour). Do not change this to a bubble-phase or stopPropagation pattern.
7. **Replay state is local-only.** Version is `1`. Bump the version when tour content changes enough that re-nudging is warranted.
8. **Info popovers are click-to-open, never hover.** Hover may carry a supplemental one-sentence label via `title` attribute.
9. **Five semantic anchor keys are the contract.** `warning-evidence`, `confidence-indicator`, `standalone-mode`, `batch-matching-logic`, `no-persistence`. Codex replaces the client fixture's content with the manifest route's content; the UI wiring by key stays identical.
10. **Nine tour target keys are the DOM contract.** Elements carrying `data-tour-target="..."` must remain in place. If Codex renames or removes a target element, update the manifest entry and/or drop the affected step.
11. **Launcher lives in the header right cluster.** No floating button, no keyboard-shortcut-only path, no hamburger menu. Matches the `TTB-105` back-button visual register.
12. **Theme tokens only.** Unchanged rule from prior stories. The `labelThumbnail.ts` SVG-content exception continues to apply (but tour code introduces no new raw hex).

## Required Codex engineering work

### Shared contract additions

Add to `src/shared/contracts/help.ts` (new file):

```ts
export const helpAnchorKeySchema = z.enum([
  'orientation',
  'intake-form',
  'processing',
  'verdict-and-checklist',
  'warning-evidence',
  'standalone-mode',
  'batch-matching-logic',
  'batch-matching',
  'no-persistence',
  'confidence-indicator'
]);

export const tourTargetKeySchema = z.enum([
  'tour-launcher',
  'tour-mode-toggle',
  'tour-drop-zone',
  'tour-verify-button',
  'tour-verdict-banner',
  'tour-warning-row',
  'tour-batch-tab',
  'tour-batch-intake',
  'tour-privacy-anchor'
]);

export const helpShowMeSchema = z.object({
  label: z.string(),
  action: z.enum(['noop', 'load-scenario', 'advance-view', 'open-panel-section']),
  payload: z.record(z.string(), z.string()).optional()
});

export const tourStepSchema = z.object({
  anchorKey: helpAnchorKeySchema,
  stepIndex: z.number().int().positive(),
  totalSteps: z.number().int().positive(),
  title: z.string(),
  body: z.string(),
  target: tourTargetKeySchema.optional(),
  interaction: z.enum(['passive', 'click-to-advance']).optional(),
  requires: z
    .object({
      scenarioId: z.string().optional(),
      expandRowId: z.string().optional(),
      variant: z.enum(['auto', 'standalone']).optional(),
      view: z.enum(['intake', 'results', 'batch-intake']).optional(),
      mode: z.enum(['single', 'batch']).optional()
    })
    .optional(),
  showMe: helpShowMeSchema.optional(),
  cta: z.string().optional()
});

export const infoPopoverSchema = z.object({
  anchorKey: helpAnchorKeySchema,
  title: z.string(),
  body: z.string()
});

export const helpManifestSchema = z.object({
  version: z.number().int().positive(),
  locale: z.string(),
  tourSteps: z.array(tourStepSchema).min(1),
  infoPopovers: z.array(infoPopoverSchema).min(1)
});
```

The UI's `src/client/helpManifest.ts` already matches this shape. Codex replaces the client fixture with a `fetch('/api/help/manifest?locale=en')` + `helpManifestSchema.parse(...)` at startup (or lazily on first tour launch / first info anchor click).

### API / server behavior

1. `GET /api/help/manifest?locale=en` — stateless, deterministic, returns `helpManifestSchema`. `Cache-Control: public, max-age=300`. No AI-generated text at runtime.
2. Optional future: `GET /api/help/recommend?view=batch-dashboard` for a context-aware first info anchor. Not required for this story; UI would surface it as a subtle pointer chip.

All endpoints honor `store: false` across the product. Help endpoints are pure read; they do not touch OpenAI at all.

### Loading, error, empty-state behavior the backend must support

- Manifest fetch failure: UI should continue to function with the client fixture (or disable help, Codex's call). Currently the client fixture is the source; cutover strategy is Codex's.
- Missing anchor content: UI renders the "Help content is on the way." fallback. Server should not return partial entries — if `tourSteps` is empty or `infoPopovers` is empty, UI treats that as a degenerate manifest and uses the fixture.
- Language: manifest includes `locale: 'en'`. Non-English locales are out of scope for this story.

### Privacy / no-persistence constraints

- No network activity on help interactions except the single manifest fetch.
- Replay state lives in `localStorage` on the reviewer's browser only. Never sent to the server.
- No reviewer identity, analytics hook, or per-user progress tracking.
- Popover opens and tour step transitions do not log anywhere persistent.

### Latency constraints

- Manifest fetch should complete in under 500 ms at the edge; UI tolerates a longer tail by rendering the launcher idle and serving tour content lazily.
- Tour step transitions are instant (React state change). Callout reposition is ≤ 250 ms.
- Info popover open/close is instant (React state change).

### Tests to derive from the accepted behavior

- Manifest shape validation: `helpManifestSchema.parse` rejects missing required fields; `tourStepSchema` enforces the `stepIndex` / `totalSteps` invariant.
- Replay state versioning: `loadHelpReplayState()` returns default when version mismatches.
- Launcher interaction: clicking launcher fires `onLaunch`, sets the active-state treatment, and dismisses the nudge.
- Tour click-to-advance: a synthetic click on the `tour-batch-tab` element advances from step 7 to step 8 AND switches mode to batch.
- Info anchor dismissal: Escape, backdrop click, and Close all dismiss; focus returns to the anchor on close.
- Accessibility: focus traps, `role="dialog"` on popovers, `aria-labelledby` on tour callouts, `aria-live="polite"` on the step indicator.
- Privacy: no `fetch` calls beyond the single manifest endpoint; no logs of tour step indices or popover opens.

## Open questions or blockers

1. **Client fixture retention.** UI ships with the full fixture in `src/client/helpManifest.ts`. Once Codex's manifest route lands, should the fixture stay as a dev/offline fallback behind an env flag, or be removed entirely? Default recommendation: keep behind an env flag so the client still works during local development if the server is offline.
2. **`GuidedReviewPanel.tsx` cleanup.** The earlier side-panel implementation is still in the codebase but unused. Codex may delete it as part of manifest cutover, or leave it as a reference. Default: delete.
3. **Tour step 5 target falling back to center.** The Verdict banner only exists on Results view; if the reviewer reaches step 5 without having run verification, the callout centers. That's the graceful fallback. If Codex wants a stricter prerequisite enforcement (e.g., block Next on step 4 until verification completes), flag it; the UI's current default is permissive — the reviewer can always skip forward via Next.
4. **Tour step 6 warning row auto-expansion.** `Show warning defect` loads the scenario but does not auto-expand the `government-warning` row. The spotlight lands on the collapsed row; the reviewer expands it themselves. Codex could wire an `expandRowId` directive (already modeled in the `requires` schema) to have the tour auto-expand; UI's current default is manual.
5. **Manifest route caching.** UI currently loads the fixture at module import time. After manifest-route cutover, recommend a one-shot fetch with an in-memory cache, invalidated on page reload. If Codex wants per-tab refetch, add a small version-check poll.
6. **Keyboard shortcut for launcher.** UI does not wire a shortcut to avoid conflicts with browser find (`/`, `?`). Codex may later add an unshifted key (e.g., `g` outside input fields) if user testing shows demand.
7. **`TTB-107` integration.** The newly-planned mock Treasury auth story will introduce a signed-in shell. When `TTB-107` lands, the `HelpLauncher` should move inside the signed-in header (not the pre-auth Screen 0). The `data-tour-target="tour-launcher"` attribute moves with it. Otherwise no tour changes are required — the signed-in shell is additive. Flagged here so the TTB-107 handoff can hand this back to Claude if the move is non-trivial.
